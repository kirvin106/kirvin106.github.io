Protocolo HTTP — Investigación

ESCAMILLA CASTILLO IRVIN ALFREDO

Resumen breve del protocolo HTTP: estructura, funcionamiento, métodos de petición y códigos de respuesta.

Introducción

HTTP (Hypertext Transfer Protocol) es un protocolo de la capa de aplicación usado para la transferencia de recursos (HTML, imágenes, JSON, etc.) entre clientes (por ejemplo, navegadores) y servidores. Es un protocolo cliente-servidor y, en su diseño clásico, sin estado.

Estructura y funcionamiento

HTTP intercambia mensajes entre cliente y servidor. Existen dos tipos de mensajes: peticiones (requests) y respuestas (responses).

Estructura básica de una petición HTTP

Request-Line
Header1: valor
Header2: valor

[cuerpo opcional]

Ejemplo de request-line: GET /index.html HTTP/1.1. Los encabezados (headers) transmiten metadatos como Host, User-Agent, Content-Type o Content-Length.

Estructura básica de una respuesta HTTP

HTTP/1.1 200 OK
Header1: valor
Header2: valor

[cuerpo opcional con el recurso solicitado]

La respuesta incluye la línea de estado (versión HTTP + código + frase) seguida de encabezados y el cuerpo. HTTP/2 y HTTP/3 difieren en el transporte/subyacente, pero la semántica de métodos y códigos se mantiene.

Métodos de petición (principales)

Los métodos indican la intención del cliente. A continuación los más usados:

  • GET — Recuperar un recurso. Seguro (no debería modificar estado) y cacheable.
  • HEAD — Igual que GET pero sin cuerpo (solo encabezados). Útil para comprobar metadatos.
  • POST — Enviar datos al servidor (por ejemplo, formularios). No es idempotente por defecto.
  • PUT — Reemplazar o crear un recurso en la URI dada. Es idempotente.
  • PATCH — Aplicar modificaciones parciales a un recurso existente.
  • DELETE — Eliminar un recurso. Normalmente idempotente.
  • OPTIONS — Consultar qué métodos están permitidos en el servidor/URI (útil en CORS y pruebas).

Conceptos relacionados: seguro (safe), idempotente y cacheable (si la respuesta puede guardarse).

Códigos de respuesta HTTP

Los códigos de estado indican el resultado de la petición. Se agrupan por clases:

  1. 1xx — Informacionales (100 Continue, 101 Switching Protocols).
  2. 2xx — Éxito (200 OK, 201 Created, 204 No Content).
  3. 3xx — Redirecciones (301 Moved Permanently, 302 Found, 304 Not Modified).
  4. 4xx — Errores del cliente (400 Bad Request, 401 Unauthorized, 403 Forbidden, 404 Not Found).
  5. 5xx — Errores del servidor (500 Internal Server Error, 502 Bad Gateway, 503 Service Unavailable).

Ejemplos comunes y significado

  • 200 OK: La petición tuvo éxito; el recurso se devuelve en el cuerpo (si procede).
  • 201 Created: Recurso creado (por ejemplo, tras un POST o PUT).
  • 301 Moved Permanently: Redirección permanente a otra URL.
  • 302 Found: Redirección temporal.
  • 304 Not Modified: El recurso no ha cambiado; el cliente puede usar la versión en caché.
  • 400 Bad Request: Solicitud malformada.
  • 401 Unauthorized: Requiere autenticación.
  • 403 Forbidden: El cliente no tiene permiso para acceder al recurso.
  • 404 Not Found: Recurso no encontrado en el servidor.
  • 500 Internal Server Error: Error genérico del servidor al procesar la solicitud.

Notas adicionales

- HTTP originalmente usa TCP como transporte (HTTP/1.1 y HTTP/2 sobre TCP); HTTP/3 usa QUIC (sobre UDP).
- HTTP es extensible: encabezados personalizados y nuevas cabeceras permiten negociaciones (por ejemplo, Accept, Content-Type).
